# Project Architecture ‚Äî WebCuaMeApplication

## Overview

A school management system with two interaction channels:

- **Admin Web** ‚Äî management dashboard via REST API (FastAPI)
- **Telegram Bot** ‚Äî direct interaction channel for teachers and students

Both channels share a single **MySQL database** and run in a **single Python process** (the bot runs via `asyncio.create_task` inside FastAPI's lifespan).

## Evaluation

### What Works Well

**1. Clean database design**
- Merging Teacher and Student into a single `users` table with a `role` discriminator reduces JOINs and simplifies `get_or_create_user`.
- `subject_classes` acts as a proper three-way junction table (subject + class + teacher) with a unique constraint.
- `ON DELETE CASCADE` and `ON DELETE SET NULL` are applied correctly throughout.
- `StudentPoint` records individual events rather than a running total ‚Äî full history is preserved.

**2. Clear separation between transport and business logic**
- `bot_handlers.py` handles only Telegram message I/O.
- `services.py` contains all DB logic and can be tested independently.
- Admin API follows a clean three-layer pattern: router ‚Üí service ‚Üí schema.

**3. Robust AI grading with proper fallbacks**
- Detects scanned-image PDFs (`__AI_READ_FAILED__`) and skips overwriting existing `ai_score`.
- Gracefully handles missing `OPENAI_API_KEY` with a user-friendly Vietnamese message instead of crashing.
- Uses upsert logic (overwrite old submission + delete old file) instead of creating duplicates.

**4. Smart teacher whitelist mechanism**
- `tele_teacher_infos` is decoupled from `users` ‚Äî admin adds telegram_id before the teacher sends `/start`.
- `get_or_create_user` automatically upgrades a student to teacher if their telegram_id is later added to the whitelist.

**5. Production-ready Docker Compose**
- Dedicated backup service running `mysqldump` via cron.
- MySQL healthcheck ensures backend waits for the database before starting.
- `mysql/init` volume auto-runs SQL on first `docker up`.

**6. Safe manual migrations in `database.py`**
- `_ensure_*` helper functions use `ALTER TABLE` to add new columns to existing production databases, avoiding `CREATE TABLE` failures on already-provisioned deployments.

---

### What Needs Improvement

**1. No authentication on the Admin API** *(Critical)*
- `ADMIN_API.md` notes that JWT auth should be added, but it has not been implemented.
- All `/admin/*` endpoints are fully open ‚Äî anyone who knows the URL can read and write all data.
- The `admin_infos` model exists in the schema but is never used by any API route.

**2. Timezone inconsistency between `models.py` and `services.py`** *(High)*
- `models.py` defaults use `VN_TZ` (UTC+7) for `created_at` columns.
- `services.py` uses `datetime.now(timezone.utc)` (UTC) in several places (e.g., `list_active_assignments_for_student`, `upsert_submission_for_student`).
- This causes incorrect deadline comparisons: a UTC+7 deadline stored in the DB is compared against a UTC `now`, making assignments appear expired 7 hours early.

**3. `app/models.py` is a stale old version** *(Medium)*
- Still contains the old `TeacherInfo` and `StudentInfo` as separate models (pre-refactor schema).
- Not imported anywhere, but causes confusion about which file is the source of truth.
- Should be deleted or moved to `backup/`.

**4. `total_students` in `ClassInfo` is denormalized and can drift** *(Medium)*
- Only incremented (+1) when a join request is approved; never decremented when a student leaves or is deleted.
- No trigger or sync mechanism exists to correct drift.
- Should be computed via `SELECT COUNT(*)` from `users` or reconciled by a background task.

**5. No structured logging** *(Medium)*
- `library.py` imports `logging` but it is not wired anywhere meaningful.
- Errors from OpenAI, Telegram, and the database are swallowed by `except Exception: pass` or silent returns ‚Äî very difficult to diagnose in production.

**6. Unvalidated file uploads** *(Low)*
- The bot accepts any file sent by a student without checking MIME type or file size.
- `safe_path_component` sanitizes the filename but there is no cap on total disk usage.

**7. Missing relationship loading after `create_subject_class`** *(Low)*
- `app/service/admin.py:163` ‚Äî after `db.commit()` and `db.refresh(record)`, the `subject_info`, `class_info`, and `teacher_info` relationships are not eagerly loaded.
